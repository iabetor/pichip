# Technical Design

## Context

Pichip 当前是一个 K 线形态相似度匹配工具，只支持价格维度的匹配。本次变更涉及 6 个新功能模块，需要考虑：
- 数据库扩展（新增 match_records 表）
- 计算性能（量能相似度增加计算量）
- 定时任务可靠性
- LLM 集成的灵活性

## Goals / Non-Goals

**Goals:**
- 提高匹配精度（加入量能维度）
- 支持完整的投资决策流程（记录→验证→分析）
- 简化用户操作（自然语言输入）
- 自动化数据维护（定时同步）

**Non-Goals:**
- 不实现 Web 界面（V3.0 规划）
- 不实现推送通知（V3.0 规划）
- 不支持多数据源（V3.0 规划）

## Decisions

### D1: 量能相似度计算方式
- **决策**: 使用换手率 + 量比，而非绝对成交额
- **理由**: 换手率消除市值差异，量比消除时间波动差异
- **公式**: `综合相似度 = 价格相似度 × (1-w) + 量能相似度 × w`

### D2: 定时任务实现
- **决策**: 使用 APScheduler
- **理由**: 纯 Python 实现，无需额外依赖，支持多种触发器
- **备选**: cron（需要系统配置）、Celery（过于复杂）

### D3: 匹配记录存储
- **决策**: 使用 SQLite 扩展表
- **理由**: 与现有数据存储一致，无需引入新数据库
- **字段**: 包含目标、匹配、相似度、过滤条件、后续走势

### D4: LLM 集成方式
- **决策**: 支持多种提供商（OpenAI/本地模型），通过配置切换
- **理由**: 用户可能有不同偏好和预算
- **实现**: 抽象 LLM 接口，具体实现可插拔

### D5: 形态识别架构
- **决策**: 基类 + 具体形态子类
- **理由**: 便于后续扩展更多形态（W底、头肩顶等）
- **结构**: `BasePattern` 定义接口，各形态继承实现

## Risks / Trade-offs

### R1: 量能相似度计算性能
- **风险**: 量能相似度计算增加约 30% 耗时
- **缓解**: 保持快速筛选（价格相关系数）作为前置过滤

### R2: akshare 数据源不稳定
- **风险**: 同步任务可能失败
- **缓解**: 增加重试机制，记录缺失数据，下次全量同步时补充

### R3: LLM 解析不准确
- **风险**: LLM 可能误解用户意图
- **缓解**: 提供反馈机制，用户可纠正解析结果

### R4: 定时任务单点故障
- **风险**: 如果调度进程崩溃，任务无法执行
- **缓解**: 记录任务执行日志，支持手动触发补偿

## Migration Plan

### 阶段一：无破坏性变更
- 量能相似度：默认 `--volume-weight 0`，不影响现有行为
- 形态记录：新增表，不影响现有数据

### 阶段二：数据迁移
- 首次运行时自动创建 `match_records` 表
- 无需迁移现有数据

### 阶段三：配置文件
- 创建 `config.yaml` 模板
- 如不存在则使用默认配置

## Open Questions

1. **量能权重默认值**: 建议 0.3，是否需要用户调研确认？
2. **LLM 接口是否必需**: 可否作为可选功能，不配置则跳过？
3. **定时任务持久化**: 是否需要支持任务执行历史查询？
